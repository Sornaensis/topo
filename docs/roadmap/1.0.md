# Topo 1.0 Readiness Roadmap

This document defines the 1.0 readiness plan for Topo. It is the checked-in
reference for 1.0 release scope, gates, sequencing, and blockers. The matching
hmem project tree is used for live task execution state, dependency tracking,
and implementation handoff notes.

## North-star outcome

Topo 1.0 should feel like a cohesive platform rather than a research
prototype:

1. `topo` remains the stable deterministic terrain, simulation, overlay,
   persistence, and plugin protocol library core.
2. `topo-seer` becomes the authoritative runtime host: SDL UI, actor system,
   direct HTTP/OpenAPI server, plugin supervision, and shared service layer.
3. Plugins are isolated OS processes with explicit manifests, dependency
   resolution, capability gates, data-resource CRUD, and backend-neutral
   external data-source declarations/status/diagnostics.
4. Plugin production transport is explicit and cross-platform: Windows named
   pipes and Unix domain sockets using length-prefixed JSON envelopes.
5. MCP is removed after direct topo-seer HTTP reaches feature parity.
6. UI behavior and HTTP behavior flow through the same service layer so state
   changes do not fork across command handlers, UI handlers, and API handlers.
7. Tests, CI, generated contracts, and docs are strong enough that public API,
   protocol, UI-state, persistence, plugin lifecycle, and HTTP behavior cannot
   drift silently.

## Non-negotiable external data-source policy

Topo must not privilege any particular backend as the shared plugin ecosystem
data model.

- External data sources belong to provider plugins, adapters, users, or
  deployment environments.
- Topo may broker backend-neutral declarations, grants, dependency/status
  information, capability checks, error reporting, and UI/API diagnostics.
- Topo must not prescribe backend-specific provisioning, migrations, locking,
  schemas, filesystem layouts, or consistency policy as part of the core spec.
- SQLite may be used as a non-normative example, fixture, or adapter, but it is
  not the privileged 1.0 shared-state model.

## Milestones

### M0 — 1.0 definition and repository hygiene

Foundational documentation and decisions before feature rewrites.

Deliverables:

- 1.0 release gates and release blockers.
- Current and target architecture documentation.
- Public module and feature inventory.
- Command method inventory.
- Current test and coverage baseline.
- Explicit decisions for HTTP framework, optional API package split, and plugin
  protocol/manifest versioning.

Acceptance:

- Every exposed module and existing command method is inventoried.
- CI work starts before major service/API/plugin/UI rewrites.
- Foundational decisions are concrete enough for implementers to avoid guessing.

### M1 — CI, coverage, and test fixtures

Build the testing foundation that protects later refactors.

Deliverables:

- Linux and Windows GitHub Actions workflows.
- SDL2/SDL2_ttf setup for CI and headless-friendly testing.
- Stack build/test/coverage jobs and coverage artifacts.
- Coverage threshold and waiver enforcement.
- Plugin fixture harness with echo, generator, simulation, CRUD, external
  data-source provider/consumer, bad-handshake, slow, and crashy fixtures.
- Headless/test-mode skeleton for `topo-seer`.
- Generated command coverage inventory test.

Acceptance:

- CI runs on pull requests.
- Linux and Windows builds pass or expose explicit tracked blockers.
- Coverage reports are generated and threshold/ratchet checks exist.
- Plugin fixture executables can be built and spawned in tests.

### M2 — Service layer extraction

Create a typed behavior boundary shared by command IPC, UI, HTTP, and tests.

Deliverables:

- `Seer.Service.AppService` plus focused services for state, config, world,
  terrain, editor, pipeline, plugins, data resources, simulation, logs,
  screenshots, UI state, and async status.
- Reusable typed errors independent of the legacy command response envelope.
- `Seer.Command.Dispatch` reduced to a thin adapter over services.
- Direct service tests for every public operation.

Acceptance:

- Existing command-dispatch behavior remains covered.
- Every command method maps to a tested service operation or explicit waiver.
- UI and HTTP can share service behavior for state-changing operations.

### M3 — HTTP and OpenAPI

Expose direct `topo-seer` HTTP over the service layer.

Deliverables:

- HTTP dependencies and package/module structure.
- Typed API routes, DTOs, errors, auth, events, and OpenAPI generation.
- Configurable loopback HTTP server with optional bearer token and non-loopback
  protection.
- Headless/test mode HTTP startup.
- Route slices for health, version, OpenAPI, events, state/UI, config, camera,
  overlays, worlds/generation, terrain, editor, pipeline, plugins, data
  resources, simulation, logs, presets, and screenshots.
- HTTP contract tests and OpenAPI drift tests.

Acceptance:

- `GET /health` works before a world is generated.
- `GET /openapi.json` returns valid generated OpenAPI.
- Every implemented route has a handler and every handler appears in OpenAPI.
- Mutating routes have validation/error tests.

### M4 — MCP migration and removal

Retire MCP after HTTP parity.

Deliverables:

- MCP-to-HTTP parity matrix.
- MCP test coverage ported to HTTP/service/OpenAPI tests.
- Docs and examples updated to HTTP/OpenAPI.
- `topo-mcp` removed from the workspace after parity.
- User-facing MCP terminology removed from command/API logs where it no longer
  applies.

Acceptance:

- Workspace builds without `topo-mcp`.
- No required 1.0 functionality depends on MCP.
- External automation docs use HTTP/OpenAPI.

### M5 — Plugin supervisor and production transport

Make plugin lifecycle and transport explicit, observable, and cross-platform.

Deliverables:

- Focused supervisor components split from the current plugin manager.
- Observable lifecycle state machine and diagnostics.
- Unix domain socket and Windows named-pipe production transports.
- Plugin launch environment contract.
- Request correlation IDs, frame limits, timeouts, heartbeat/health, restart
  policy, and failure handling.
- Detailed plugin status in UI/API.
- Supervisor and transport tests using fixture plugins.

Acceptance:

- Plugins run over real platform transports in CI.
- Stdio is test/development compatibility only if it remains at all.
- Bad plugins fail with actionable diagnostics instead of hanging topo-seer.

### M6 — Manifest v3, dependency resolver, and CRUD hardening

Make plugin declarations, dependency ordering, and data-resource behavior stable.

Deliverables:

- Manifest v3 types, schema, golden examples, generated docs, and SDK output.
- Host validation with actionable errors.
- Unified dependency graph covering built-in stages, plugins, overlays,
  resources, backend-neutral external data-source provider/consumer
  relationships, capabilities, and optional/soft dependencies.
- Deterministic startup, pipeline insertion, and simulation DAG order.
- Versioned data-resource schemas, pagination, sorting/filtering, etags,
  standardized errors, and host validation of inbound/outbound records.

Acceptance:

- Missing/cyclic dependencies are diagnosed before startup.
- CRUD behavior is consistent across service, UI, HTTP, and plugin boundaries.
- No manifest/dependency/CRUD contract privileges a specific external backend.

### M7 — Backend-neutral external data sources

Represent plugin ecosystem data dependencies without making topo own the data
backend.

Deliverables:

- Backend-neutral external data-source declarations and grants.
- Opaque data-source reference/configuration model.
- Provider-owned migration/schema/consistency boundaries.
- Backend-neutral health/status/policy metadata.
- Access grant and failure cleanup policy.
- Provider/consumer fixtures and integration tests.
- UI/API diagnostics for requirements, grants, provider relationships, health,
  and failure reasons.

Acceptance:

- No part of the spec privileges SQLite or any other backend.
- Topo does not own plugin-ecosystem external data stores.
- Provider/consumer plugins coordinate through declared capabilities, status,
  diagnostics, and dependency ordering.
- Persistence stores references/metadata where appropriate, not backend internals.

### M8 — UI componentization

Decompose `topo-seer` UI into testable components.

Deliverables:

- `Seer.System` split into boot, event pump, render frame, snapshot/cache, and
  screenshot helpers.
- `Seer.Draw.Config` split into focused panels.
- Typed geometry/layout modules.
- Component reducers, stable widget IDs, normalized input events, and draw
  commands.
- Standalone DataBrowser component using AppService.
- Layout, reducer, draw-command, and headless smoke tests.

Acceptance:

- Config, data, pipeline, plugin, and simulation panels are separate modules.
- DataBrowser has pure reducer tests.
- Mutating UI behavior shares service semantics with HTTP.

### M9 — Feature completion matrix

Expose every important library domain through UI and service/API where useful.

Deliverables:

- Maintained feature matrix checked in CI.
- Canonical terrain inspector with sections for coordinates/chunk,
  elevation/hypsometry, tectonics, erosion/terrain form, hydrology/water,
  climate/weather, biome/soil/vegetation, glacier, volcanism, ocean currents,
  overlays, plugin records, provenance, and units.
- View mode registry, legends, and export mappings.
- Missing domain UI/API surfaces.
- Pipeline/stage diagnostics.
- Overlay/schema/provenance UI and export/import dialogs.
- Simulation DAG and plugin resource/external data-source status surfaces.

Acceptance:

- Every library feature group has UI surface or approved waiver.
- Useful surfaces have service/API coverage.
- No unplanned `Missing` rows remain in the maintained feature matrix.

### M10 — Release hardening, docs, and packaging

Finalize the 1.0 release.

Deliverables:

- README and runtime docs updated for 1.0.
- Generated/published OpenAPI documentation.
- Plugin manifest, RPC, pipeline, and external data-source docs expanded or
  generated.
- Migration guide and MCP removal notes.
- Package versions, changelog, release notes, tags, and packaging instructions.
- Full CI, performance, HTTP, plugin, and world smoke checks.
- Final release blocker audit.

Acceptance:

- CI passes.
- Docs and release notes are complete.
- Plugin/API compatibility policy is documented.
- No MCP remains as required 1.0 functionality.
- No critical feature gaps or release blockers remain.

## Suggested pull request sequence

1. Add 1.0 roadmap and current/target architecture docs.
2. Add CI workflows and coverage reporting.
3. Add feature and command inventory scripts.
4. Add headless/test-mode skeleton.
5. Extract AppService for read-only state/config/world queries.
6. Extract AppService for mutations and typed errors.
7. Add HTTP health/version/openapi/state routes.
8. Add HTTP config/world/terrain routes.
9. Add HTTP plugin/data/simulation routes and events.
10. Port MCP tests to HTTP.
11. Remove MCP package and docs.
12. Add plugin supervisor lifecycle and production transport.
13. Add manifest v3 and dependency resolver.
14. Add backend-neutral external data-source declarations/grants/status.
15. Refactor DataBrowser and focused config/pipeline/plugin panels.
16. Add missing feature UI/API surfaces.
17. Complete release docs, packaging, and final smokes.

## Per-PR definition of done

Every PR touching runtime behavior should include:

- Tests for new behavior.
- Error-path tests for validation/failure cases.
- Coverage that does not drop below threshold once thresholds exist.
- Updated docs and generated/golden specs for public API, HTTP, or protocol
  changes.
- UI layout/reducer tests for UI behavior changes where practical.
- Plugin fixture or protocol tests for plugin behavior changes.
- Feature/test matrix updates for new exposed modules.
- No public behavior bypassing AppService unless explicitly justified.

## Hard 1.0 blockers

- Any package fails to build on Linux or Windows.
- OpenAPI route/handler/schema drift exists.
- Required functionality still depends on MCP.
- Production plugin transport relies on undocumented stdio behavior.
- Any external data-source contract privileges a backend or makes plugin
  ecosystem data storage topo-owned.
- UI lacks a path to inspect major library domain data.
- World save/load cannot preserve plugin overlays and external data-source
  references/metadata where applicable.
- Plugin crashes can hang topo-seer or corrupt app state.
- Coverage thresholds are not enforced.
- Manifest, RPC, pipeline, or external data-source docs remain stubs.
