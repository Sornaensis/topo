# Migrating to topo-seer 1.0

Version 1.0 removes transitional controls, transport surfaces, route aliases,
and ambiguous view/export names. Removed HTTP paths are not aliases in 1.0;
they return 404. Use the live `GET /openapi.json` contract after migration.

## Config snapshots

The current `ConfigSnapshot` version is `2`.

- A snapshot tagged below version 2 loads with current defaults for missing
  fields and is reported as version 2.
- A future version tag is retained so a caller can identify data written by a
  newer release.
- Unknown extra object keys are ignored. Current serialization does not write
  removed keys.

Make these changes to older `genConfig` objects:

| Remove | 1.0 treatment |
|---|---|
| Haskell `pcDetailScale` / JSON `detailScale` | Ignored on read; no replacement. Retained parameter values still load. |
| Climate `boundary` and children such as `motionTemp` | Ignored on read; no replacement. Retained climate objects such as `temperature` still load. |
| Snapshot view keys such as `view`, `view_mode`, `viewMode`, `viewSelection`, `weather_basis`, or `overlay_opacity` | Ignored on read and not serialized. Select the layered view at runtime. |

Do not keep writing these keys: their values have no effect. Loading a named
world resets to the normal default layered view. Loading a preset applies
configuration but does not restore or change the current view selection.

## Replace the command socket transport

The old command socket and its command-envelope methods are not public 1.0
interfaces. There is no `/commands` HTTP resource. Migrate external automation
to the resource-oriented HTTP API and discover it through
`GET /openapi.json`. In-process integrations should call the focused operations
on `Seer.Service.AppService` instead of constructing command envelopes.

In particular, removed `get_view_modes` and `set_view_mode` methods are unknown;
use the layered resources described below.

## Use canonical HTTP resources

The left column lists removed pre-1.0 requests. Each returns 404 in 1.0.

| Removed request | Canonical 1.0 request |
|---|---|
| `GET /terrain/overlays` | `GET /overlays` |
| `POST /ui/overlay` | `PUT /overlays/current` |
| `PUT /ui/overlay` | `PUT /overlays/current` |
| `GET /ui/overlay-fields` | `GET /overlays/fields` |
| `POST /ui/overlay/cycle` | `POST /overlays/cycle` |
| `POST /ui/overlay-field/cycle` | `POST /overlays/fields/cycle` |
| `GET /ui/camera` | `GET /camera` |
| `PUT /ui/camera` | `PUT /camera` |
| `POST /ui/camera/zoom-to-chunk` | `POST /camera/zoom-to-chunk` |
| `GET /plugins/status` | `GET /plugins` |
| `GET /plugins/state` | `GET /plugins` |
| `GET /plugins/dependencies` | `GET /plugins` |
| `GET /state/view-modes` | `GET /state/views` |
| `POST /ui/view-mode` | `POST /ui/view` |

Update request schemas as well as paths; the current OpenAPI document defines
the accepted bodies.

## Replace a single view with layers

A single `mode` no longer represents the renderer state. Query
`GET /state/views`, then send the fields that should change to
`POST /ui/view`:

```json
{
  "base_mode": "elevation",
  "overlay_mode": "cloud",
  "weather_basis": "average",
  "overlay_opacity": 0.6
}
```

The canonical model has an independent `base_mode`, nullable `overlay_mode`,
optional `plugin_overlay` and `field_index`, `weather_basis`, and
`overlay_opacity`. Opacity is clamped to `[0,1]`; plugin overlay names and field
indexes are validated.

## Qualify terrain export fields by basis

The terrain exporter rejects the ambiguous names `temperature`,
`weather_temperature`, and `normal_temperature` with structured
`unknown_field` validation. Choose the intended temporal basis explicitly:

| Intended data | Canonical field |
|---|---|
| Generated long-run temperature average | `climate_temp_avg` |
| Current simulated temperature | `weather_temp_current` |
| Generated typical temperature normal | `weather_temp_typical` |

The same naming rule applies to supported related values. For example, use
`climate_precip_avg`, `weather_precip_current`, or
`weather_precip_typical`; current pressure is
`weather_pressure_current`; cloud values include
`weather_cloud_cover_current` and `weather_cloud_cover_typical`.
The service's `available_fields` list is the authoritative set for the loaded
world.

## Runtime availability

Screenshot success now means capture by the SDL renderer. `POST /screenshots`
in a headless runtime returns structured HTTP 503 `unavailable` with
`screenshot capture requires the SDL renderer`; it does not manufacture image
bytes or write a file. Clients that can run against either mode must handle this
availability response.
