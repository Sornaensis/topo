# Topo 1.0 Release Notes

These notes summarize public-surface changes for the 1.0 release line. See the
[pre-1.0 migration guide](../migration/pre-1.0-to-1.0.md) for step-by-step
client and plugin upgrade guidance. Packaging, artifact, and tag instructions
live in the [release packaging guide](../release.md), with a concise history in
[CHANGELOG.md](../../CHANGELOG.md).

## Package and artifact versions

- Workspace packages are released at Cabal version `1.0.0.0`: `topo`,
  `topo-seer`, `topo-plugin-sdk`, `topo-plugin-example`, and
  `topo-plugin-civ-example`.
- The HTTP/OpenAPI contract is API version `1.0.0` with `api_version=1`; the
  `/version` route reports the `topo-seer` package version separately.
- The release tag is `v1.0.0` and covers the whole Stack workspace.
- Reproducible artifacts are source tarballs for all five packages, staged
  executables for `topo-seer` and the plugin fixture/examples, and the committed
  documentation/OpenAPI artifacts listed in `docs/release.md`.

## Automation and API compatibility

- Direct HTTP/OpenAPI served by `topo-seer` is the only public 1.0 automation
  surface.
- The former `topo-mcp` bridge has been removed. MCP stdio JSON-RPC,
  `tools/list`, `tools/call`, `resources/list`, `resources/read`, and
  `topo://` resources have no 1.0 compatibility promise.
- Legacy command IPC and command-compatible HTTP routes remain internal/test
  compatibility only while service extraction continues; they are not public
  automation paths.
- The OpenAPI document served at `GET /openapi.json` is the client operation
  catalog. API major version `1` allows additive routes, fields, enum values,
  and schemas; breaking route, payload, authentication, or error-envelope
  changes require a new major API version or a documented migration window.

## Plugin compatibility

- Manifest v3 (`manifestVersion: 3`) and RPC protocol v4 are the supported 1.0
  plugin contracts.
- Plugin manifests declare protocol bounds in `runtime.protocol.min` and
  `runtime.protocol.max`; plugins should test against protocol `4` and keep
  bounds narrow.
- Production plugin transport uses host-created Windows named pipes or Unix
  domain sockets with length-prefixed JSON envelopes. Stdio is only explicit
  test/development compatibility.
- The protocol v4 contract includes data-service CRUD payloads, request
  correlation, heartbeat/health messages, external data-source grants,
  revocations, status requests, status reports, and launch auth proof.

## External data-source compatibility

- External data-source declarations are backend-neutral and provider-owned.
  Topo brokers source/grant metadata, generic capabilities, status, diagnostics,
  and opaque references; it does not own provider storage.
- Provider plugins, adapters, deployment tooling, or external systems own
  provisioning, migrations, backing schemas, connection strings, locking,
  writer coordination, consistency policy, and repair behavior.
- SQLite may be used as a non-normative example, fixture, or optional adapter,
  but it is not a privileged Topo 1.0 shared-state model and is not a core spec
  backend.
