# ADR 0002: Plugin Protocol and Manifest Versioning Strategy

- **Status:** Accepted for 1.0 planning
- **Date:** 2026-05-14
- **Decision owners:** Topo maintainers

## Context

The current plugin foundation already includes manifests, capabilities,
length-prefixed JSON messages, handshake/world-changed notifications,
generator/simulation invocation, data-resource query/mutation messages, and an
SDK runner that generates manifests from `PluginDef`.

The current code identifies the plugin RPC protocol as version `2`. Version 2
introduced handshake exchange, world-changed notifications, and data-service
CRUD messages. Some message payloads also carry a separate `payload_version`
currently fixed at `1`.

Today the SDK `runPlugin` path generates `manifest.json` from `PluginDef` and
connects over stdin/stdout. Protocol v3 replaces production launch with
environment-provided named-pipe/Unix-domain-socket endpoints, leaving stdio only
as explicit test/development compatibility.

For 1.0, the plugin system needs larger contract changes:

- production transport moves to host-created Windows named pipes and Unix domain
  sockets rather than documented-but-not-yet-complete platform connection logic;
- stdio may remain useful for tests or development compatibility, but it should
  not be the production contract;
- plugin lifecycle, request correlation IDs, timeouts, health/heartbeat,
  capability negotiation, dependency diagnostics, and external data-source
  declarations/status need stable representation;
- manifests need a complete versioned schema generated by the SDK;
- examples need to demonstrate the same contract plugin authors are expected to
  use for 1.0.

## Decision

Define **plugin protocol v3** and **manifest v3** as the production-supported
1.0 plugin contract.

Protocol, manifest, resource schemas, and application/API versions are separate
version axes:

| Version axis | 1.0 target | Owns |
|---|---:|---|
| Plugin RPC protocol | `3` | session envelope, message types, handshake, lifecycle, transport expectations, request correlation, timeout/error semantics |
| Plugin manifest schema | `3` | plugin identity, runtime, capabilities, params, generator/simulation/resource declarations, dependency declarations, external data-source declarations, UI hints |
| Data-resource schema | independently versioned | CRUD schemas, operations, pagination/filter/sort, conflict fields, validation rules |
| HTTP/API | independently versioned | topo-seer public HTTP routes and OpenAPI |

The SDK-generated manifest is the source of truth for plugin declarations.
Manual manifest editing may remain possible for advanced users, but the SDK must
be able to generate a complete manifest v3 for all first-party examples.

## Production-supported 1.0 plugin contract

A plugin is considered production-supported for 1.0 only when it uses:

1. manifest v3 generated or validated against the v3 schema;
2. protocol v3 handshake and message envelopes;
3. host-created named pipe or Unix domain socket endpoint;
4. endpoint/session/auth metadata passed through launch environment variables;
5. length-prefixed JSON framing;
6. request correlation IDs for request/response-style messages;
7. negotiated protocol/manifest version and feature set;
8. capability gates enforced by the host;
9. lifecycle/status/error reporting visible to UI and HTTP;
10. backend-neutral external data-source declarations/status when a plugin uses
    ecosystem-shared external data.

## Compatibility policy

Compatibility exists only to make migration safe; it is not the 1.0 public
contract.

### Current protocol and manifest compatibility

- The host may temporarily read current pre-v3 manifests during migration if it
  can produce clear deprecation diagnostics.
- The host may temporarily speak protocol v2 to existing first-party examples
  only while v3 support is being implemented.
- All first-party examples must be upgraded to manifest v3 and protocol v3
  before 1.0 release.
- 1.0 docs should describe manifest v3 and protocol v3 as the supported plugin
  contract. Older formats may be documented only in migration notes.

### Stdio compatibility

- Stdio transport may remain behind an explicit test/development compatibility
  mode.
- Stdio must not be used in production plugin launch documentation for 1.0.
- Tests should prove production plugin launch uses endpoint metadata and real
  platform transports, not implicit stdio pipes.

### Compatibility fields

- Temporary compatibility fields may be emitted by the SDK only when marked as
  deprecated and covered by tests.
- Compatibility fields should have a removal milestone or documented 1.0 waiver.
- Compatibility fields must not be required for a clean manifest v3 or protocol
  v3 implementation.

## Versioning rules

### Protocol versions

- Increment the protocol major integer when required message semantics change in
  a way older peers cannot safely ignore.
- Add optional fields or optional message groups only behind feature negotiation
  when older peers can safely ignore them.
- Every request/response-style protocol v3 message should carry a correlation
  ID.
- Handshake must negotiate protocol version, manifest version, feature flags,
  maximum frame size, timeout expectations, resource schema versions, and
  external data-source capabilities/status support.

### Manifest versions

- `manifest_version` is required in manifest v3.
- Unknown required sections fail validation with actionable diagnostics.
- Unknown optional sections may be ignored only when the manifest marks them as
  optional or feature-gated.
- The host validates executable/runtime declarations, transport support,
  protocol min/max, capabilities, params, generator/simulation declarations,
  resources, dependencies, external data-source declarations, and UI hints before
  startup.

### Data-resource schema versions

- Resource schema evolution is independent of manifest/protocol version.
- Resource schemas should declare their schema/resource version and supported
  operations.
- CRUD contracts should support stable error codes, validation errors,
  pagination, filtering/sorting declarations, and conflict/version fields.

## SDK generation policy

The SDK should generate manifest v3 from `PluginDef` and related SDK types.

Required SDK behavior:

- Generate `manifest_version: 3`.
- Generate runtime/protocol bounds and supported transports.
- Generate capabilities from explicit declarations or safe inference.
- Generate generator, simulation, overlay, resource, dependency, and external
  data-source declarations.
- Refuse to generate invalid v3 manifests without actionable errors.
- Include golden tests for generated example manifests.
- Keep runtime message dispatch aligned with protocol v3.

The SDK should not make backend-specific external data-source assumptions. It
may expose generic provider/requirement/reference types that adapters or plugins
specialize outside topo core.

## Example plugin upgrade policy

First-party examples are part of the compatibility story and should be upgraded
as part of the v3 work:

- `topo-plugin-example` remains the minimal generator fixture using manifest v3
  and protocol v3.
- `topo-plugin-civ-example` becomes the richer full-stack fixture covering
  overlay, generator, simulation, data resources, dependency declarations, and
  backend-neutral external data-source provider/consumer behavior where useful.
- Failure fixtures should cover bad manifest version, protocol mismatch,
  unsupported transport, timeout, crash, malformed frame, and unavailable
  external data-source provider.

## Testing requirements

Protocol and manifest v3 work must include tests for:

- manifest v3 JSON round trips and golden examples;
- manifest validation errors with actionable diagnostics;
- protocol v3 envelope round trips;
- malformed frame, oversized frame, invalid JSON, and unknown message handling;
- handshake version negotiation and mismatch failure;
- feature negotiation;
- request correlation and timeout behavior;
- real Unix domain socket transport on Linux/macOS;
- real Windows named-pipe transport on Windows;
- stdio compatibility mode only where explicitly enabled for tests;
- upgraded first-party example plugins.

## Documentation requirements

Before 1.0, docs must include:

- manifest v3 field reference;
- protocol v3 envelope and message reference;
- transport launch contract and environment variables;
- version negotiation and feature flag rules;
- migration notes from current protocol/manifest behavior;
- clear statement that stdio is not the production transport;
- backend-neutral external data-source declaration/status semantics.

## Rejected alternatives

### Continue extending the current version 2 contract without a new major version

The 1.0 changes are broad enough that silent extension would make compatibility
unclear. A new major protocol and manifest version gives implementers a clear
boundary.

### Treat manifest and protocol as one version

Manifest evolution and session protocol evolution change for different reasons.
Keeping them separate lets resource schemas, manifest fields, and wire messages
evolve independently with clearer diagnostics.

### Keep stdio as the documented production transport

Stdio is convenient for tests and simple process launch, but it conflicts with
the intended isolated endpoint model and makes endpoint/session/auth metadata
less explicit. It remains acceptable only as an explicit compatibility mode.

### Encode a particular external data backend in manifest v3

Plugin ecosystems may use many external data sources. The manifest should
describe backend-neutral requirements, provider relationships, grants, and
status expectations rather than making any backend part of topo's core plugin
contract.
